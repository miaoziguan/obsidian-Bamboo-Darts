# 维度一：产品设计与用户体验 — Bamboo Darts（竹叶飞刃）

> 产品类型初判：AI 驱动的效率/工具类产品，寄生于 Obsidian 生态。技术密度极高，本维度重点拆解"为什么这样设计"而非"好不好用"。

## 一、核心价值主张与问题定义

Bamboo Darts 解决的核心问题，不是"怎么记笔记"，而是**"怎么阻止无效信息进入你的知识库"**。

现代知识工作者的真实困境不是缺信息，而是信息过载：浏览器里躺着 2000+ 未读收藏，Notion 里 37 个未完成项目，Obsidian 库里堆着大量剪藏却从不回看的长文。传统的"剪藏—存档"工作流（Readwise、Web Clipper、Cubox）本质是**信息囤积**——把垃圾从一个地方搬到另一个地方，问题原封不动。

Bamboo Darts 的价值主张一句话说清：**"把阅读转化为可检索的知识节点，并在入库前把信息垃圾挡在门外。"** 它不是剪藏工具，而是一台**知识精炼厂**——原料（长文/网页/选中文本）进去，经过七道工序，出来的是符合"原子笔记五条标准"的、去过重的、核查过事实的知识节点。

这个定位的独特性在于**"做减法"而非"做加法"**。市面上绝大多数 AI 笔记插件（Copilot、Text Generator）是在帮你**生成更多内容**，而 Bamboo Darts 是在帮你**筛掉更多内容**——它的质量门控（Phase 2）会硬阻断低质输入，笔记复查（Phase 6）会过滤掉低价值输出，两端收口。在一个"AI 让内容生产成本趋近于零、信息过载空前严重"的时代，谁能帮用户做减法，谁就抓住了真痛点。

PMF 程度判断：**产品与需求高度契合，但需求本身是"高认知门槛"需求**。真正理解"原子笔记"和"Zettelkasten"价值的用户会觉得这个产品是刚需；但这类用户在 Obsidian 150 万用户中占比不高——这是它 PMF 的天花板所在（详见维度三、四）。

## 二、产品架构与设计决策

### 核心设计决策 1：七阶段流水线（Pipeline as Product）

产品最根本的设计决策，是把"AI 提炼笔记"这件看似简单的事，拆成了一条**七阶段质量流水线**：

| 阶段 | 名称 | 本质 |
|:---:|------|------|
| Phase 1 | 读取内容 | 多源输入统一化 |
| Phase 2 | 质量门控 | **入口过滤**（挡垃圾进） |
| Phase 3 | AI 提炼 | 调 DeepSeek 拆原子笔记 |
| Phase 4 | 同批去重 | BM25+分词+余弦 |
| Phase 4b | 知识库去重 | SimHash 指纹+BM25 |
| Phase 5 | 内容核查 | 三层管线查事实 |
| Phase 6 | 笔记复查 | **出口过滤**（挡垃圾出） |

这个架构的深层逻辑是：**AI 提炼（Phase 3）只是整条管线的 1/7**。真正构成产品壁垒的，是围绕 AI 前后那 6 道**非 AI 或轻 AI 的工程化质检工序**。这是一个反直觉的设计判断——在所有人都把"接入大模型"当卖点时，Bamboo Darts 把宝押在了"大模型前后的确定性工程"上。

### 核心设计决策 2：成本敏感的"零 API / 轻 API"分层

产品在多处刻意规避 AI 调用，体现出极强的成本意识：

- **内容核查 Phase 5** 设计成三层管线：Layer 1 原文溯源（**零 API**，纯字符串匹配）→ Layer 2 语义比对（**单次 AI 调用**，且只发送截断文本）→ Layer 3 超源标记（**零 API**）。绝大多数事实声明在 Layer 1 就被解决，只有少数疑难才升级到花钱的 Layer 2。
- **去重 Phase 4/4b** 全部用本地算法（BM25、SimHash、jieba 风格分词、余弦相似度），**零 API**。
- **语义去重（Beta）** 才引入向量模型（腾讯混元），且是"本地快筛 + 向量精判"的两级结构，只对可疑重复才调用向量 API。

这套设计的逻辑是：**把能用确定性算法解决的问题绝不交给 AI**。对一个依赖用户自付 API Key 的工具来说，帮用户省 token 就是核心体验——这是设计哲学层面对"用户为每次调用付费"这一商业约束的直接回应。

### 核心设计决策 3：门控的"硬阻断 + 软警告 + 累积升级 + 强制提炼"四态机

质量门控没有做成简单的"通过/拒绝"二值判断，而是一套**分级响应机制**：轻度问题软警告放行，多个软警告累积会升级为阻断，严重问题直接硬阻断，但用户始终保留"强制提炼"的逃生舱。这个设计在"帮用户把关"和"不越俎代庖"之间找到了平衡——它有主见（会拦），但不独裁（能被覆盖）。

## 三、用户体验拆解

**核心用户路径（Critical Path）**：选中文本/贴 URL → 触发提炼 → 观看七阶段实时进度 → 结果弹窗预览/勾选 → 保存入库。从触发到保存，核心动作约 3-4 步，摩擦点主要在**首次配置**（需要填 DeepSeek API Key）。

**Aha Moment**：用户第一次看到一篇 3000 字长文被拆成 5-6 条独立、精炼、已去重、标好 Tag 的原子笔记，并意识到"其中 2 条因为和库里已有笔记重复被自动拦下"——那一刻会真正理解这个产品的价值。这个 Aha 到达速度取决于用户库里是否已有存量笔记（去重才有戏），所以**老用户比新用户更容易被打动**。

**交互亮点**：七阶段**实时进度反馈**（显示当前阶段名、耗时、子进度、可随时取消）。这在 AI 类产品里是稀缺的——大多数 AI 插件是"转圈等待黑盒"，而 Bamboo Darts 把一个可能耗时 10-30 秒的 AI 流程，做成了**可观测、可中断**的过程。这背后是对"AI 等待焦虑"这一体验痛点的精准回应。

**结果弹窗（Result Modal）**是产品体验的核心承载——它不是简单展示结果，而是一个**决策界面**：展示每条笔记的去重相似度（本地 X% / 语义 Y%）、核查标记（已溯源/需对比/超源）、复查评分（差/中/良/优），让用户在保存前做最终裁决。产品把"质量信息"充分暴露给用户，体现了"辅助决策而非替代决策"的克制。

**性能体验**：发现索引（Discovery Index）缓存笔记特征（标题、关键词、内容哈希），使"关联推荐"在几千篇笔记规模下仍能秒级响应，避免每次重复读文件。这是对"知识库越大越卡"这一 Obsidian 老问题的工程化解法。

## 四、差异化与创新分析

差异化功能盘点及"核心 vs 边缘"判断：

| 功能 | 差异化程度 | 性质 |
|------|-----------|------|
| 七阶段质量管线 | 极高 | **核心体验差异** |
| 双层去重（同批+知识库） | 高 | **核心体验差异** |
| 三层内容核查 | 高 | 核心体验差异 |
| 中文分词+BM25+SimHash 本地算法栈 | 高 | 技术壁垒 |
| 质量门控四态机 | 中高 | 核心体验差异 |
| 关联推荐+MMR 重排 | 中 | 边缘增强 |
| 语义去重（Beta） | 中 | 探索性功能 |

护城河判断：**壁垒是"工程复杂度 + 中文优化"的复合体，属于设计/技术壁垒，而非数据或网络壁垒**。竞品要追平，不是接个大模型 API 那么简单，而要重建整条质检管线、调好中文分词和去重阈值——这是数月的工程投入。但这种壁垒的问题在于：**它对用户不可见**。用户很难在插件商店的一句话简介里理解"三层核查"和"SimHash 预过滤"的价值，导致优秀的工程无法直接转化为认知优势（这是产品最大的隐性矛盾，详见综合分析）。

## 五、产品设计的局限与取舍

1. **强依赖外部 AI，无法离线**——这是刻意取舍：把智能能力外包给 DeepSeek，团队专注质检工程。代价是产品命运与第三方 API 绑定。
2. **配置项爆炸**——设置页有 30+ 配置项（API/提炼/去重/标签/核查/复查/发现全都可调）。这是"给专业用户极致控制权"的选择，代价是新手门槛陡峭，与 Obsidian 本身"极客气质、学习曲线陡"的问题叠加。
3. **UI 层是工程短板**——`setting-tab.ts` 单文件 36KB、UI 层零自动化测试覆盖。团队把工程质量投入到了核心算法层（测试充分、死代码已清理），UI 层相对薄弱。这是"资源有限下的合理倾斜"，但也是产品体验进一步打磨的障碍。
4. **面向"重度中文知识工作者"的窄定位**——中文分词、DeepSeek/混元、原子笔记方法论，三重叠加把目标用户收得很窄。这是聚焦，也是天花板。

---

**产品设计评级：A**

**核心结论**：一台"做减法"的知识精炼厂——用七阶段质检管线把 AI 关进工程笼子里，工程优秀但价值对用户不可见。
